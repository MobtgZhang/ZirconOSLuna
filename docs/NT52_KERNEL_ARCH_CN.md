# Windows XP 64 位（NT 5.2）内核架构详解

Windows XP Professional x64 Edition 与 **Windows Server 2003 x64** 同属 **NT 5.2** 产品线：二者共享同一套 **64 位内核与用户态基础** 的版本脉络（具体构建号随补丁包变化）。它是微软第一款面向消费级市场的 **x86-64（AMD64）** Windows，**不是** Itanium（IA-64）版 XP。

---

## 一、整体分层结构

```
┌─────────────────────────────────────────────────────┐
│               用户模式（User Mode，Ring 3）           │
│  Win32 应用  │  WOW64 子系统  │  原生 64 位应用        │
├─────────────────────────────────────────────────────┤
│           内核模式（Kernel Mode，Ring 0）             │
│  执行体（Executive）                                  │
│  ├─ 对象管理器 / 进程管理器 / 内存管理器              │
│  ├─ I/O 管理器 / 电源管理器 / 安全引用监视器          │
│  ├─ 即插即用管理器 / 配置管理器（注册表）             │
│  微内核（Microkernel）                                │
│  硬件抽象层（HAL）                                    │
├─────────────────────────────────────────────────────┤
│               硬件层（x86-64 / AMD64）                │
└─────────────────────────────────────────────────────┘
```

---

## 二、处理器架构支持

NT 5.2 x64 原生支持 **AMD64（x86-64）** 指令集，同时也支持 Intel 的 **EM64T** 实现。值得注意的是：

* **不支持** Intel Itanium（IA-64），那是另一个独立版本（Windows XP 64-bit Edition for Itanium，基于 NT 5.1）
* 寄存器从 8 个通用寄存器扩展到 **16 个 64 位通用寄存器**（RAX、RBX、…、R8–R15）
* **虚拟地址** 在 x86-64 **长模式**下通常只使用 **低 48 位**（canonical 地址）：高位须为第 47 位的符号扩展；**中间区域**（用户上限之上、内核起始之下）为 **非规范地址**，访问会触发 fault。Windows 将 **用户态** 可用上限约束在约 **8 TB** 量级，内核使用 **高位规范** 区域（与 32 位 NT 的 2 GB/3 GB 用户上限相比，用户态空间大幅放宽）。

## 三、内存管理架构

### 3.1 虚拟地址空间布局（教学用近似）

下列区间与本仓库常量 `src/mm/va_layout.zig` 对齐，用于 **规范对照**；真实 Windows 内核还会划分 **系统 PTE、会话空间、非分页池** 等子区域，详见 *Windows Internals* 与第三方对「x64 内核 VA 布局」的分区文章。

| 区域 | 地址范围（近似） | 说明 |
|------|------------------|------|
| 用户态 | `0x00000000_00000000` – `0x000007FF_FFFFFFFF` | 约 **8 TB** 用户可映射区（本仓库 `user_space_end`） |
| 中间「空洞」 | 非规范 | 不应出现可长期映射的有效指针 |
| 内核态（高位规范） | 自 `0xFFFF0800_00000000` 起至最高规范地址 | 内核、驱动、系统结构；本仓库 `kernel_space_start` 为 **起始下界示意** |

与 32 位 NT 的 2 GB/2 GB 或 3 GB/1 GB 分割相比，**单进程用户态**可寻址范围显著增大；内核仍受 **物理内存、池大小与实现细节** 约束，并非「无限大」。

### 3.2 长模式分页与 VA 拆分（4 级页表）

AMD64 **长模式**下，常用 **4 级** 页表遍历：**PML4 → PDPT → PD → PT → 页内偏移**。虚拟地址（有效位）可理解为：

- **位 63:48**：须等于位 47 的符号扩展（canonical 规则）
- **位 47:39**：PML4 索引
- **位 38:30**：PDPT 索引
- **位 29:21**：PD 索引
- **位 20:12**：PT 索引
- **位 11:0**：页内偏移（4096 字节页时为 12 位）

本仓库在 `src/arch/x86_64/paging.zig` 中提供与上述拆分一致的 **索引掩码/移位** 常量，便于与 OS 课程或 OSDev 资料对照。

* 基础页大小：**4 KB**
* 大页（Large Page）：**2 MB**
* 巨页（Huge Page）：**1 GB**（依赖 CPU 与系统是否启用）

### 3.3 物理内存上限

NT 5.2 x64 支持最高 **128 GB** 物理内存（服务器版本支持更多），彻底摆脱了 32 位版本即使开启 PAE 也只能用 64 GB 的限制。

## 四、WOW64 子系统（32 位兼容层）

这是 NT 5.2 x64 架构中最具特色的部分之一。

```
32 位应用程序
    │
    ▼
WOW64.dll（拦截所有系统调用）
    │
    ├── wow64win.dll（Win32k 用户态调用转换）
    ├── wow64cpu.dll（CPU 状态切换，32→64 位上下文）
    │
    ▼
64 位原生 NT 内核（ntoskrnl.exe）
```

**工作原理：**

1. 32 位进程运行在独立的 **32 位虚拟地址空间**（0–4 GB）内
2. WOW64 层将 32 位的 `int 0x2E` 或 `sysenter` 系统调用**翻译**为 64 位的 `syscall` 指令
3. 文件系统重定向：32 位程序访问 `System32` 时会被透明重定向到 **SysWOW64** 目录
4. 注册表重定向：32 位程序的注册表写入被重定向至 `HKLM\SOFTWARE\Wow6432Node`

**限制：** 16 位应用程序（NTVDM）**不受支持**，因为 x86-64 长模式下无法运行 V86 模式。

## 五、内核执行体（Executive）主要组件

### 5.1 对象管理器（Object Manager）

* 所有内核资源（进程、线程、文件、事件等）统一抽象为**对象**
* 维护命名空间（`\Device\`、`\Driver\`、`\BaseNamedObjects\` 等）
* 采用**引用计数**管理对象生命周期

### 5.2 进程与线程管理器

* 每个进程拥有独立的 **EPROCESS** 结构（64 位下该结构更大）
* 线程由 **ETHREAD** 描述，调度单元为线程而非进程
* 支持**纤程（Fiber）**作为用户态协作式调度原语

### 5.3 内存管理器

* 工作集（Working Set）管理
* 页面置换（改进的 Clock 算法）
* 节对象（Section Object）用于内存映射文件和共享内存

### 5.4 I/O 管理器

* 基于**IRP（I/O Request Packet）**的异步 I/O 模型
* 驱动程序**必须为原生 64 位**，不允许加载 32 位内核驱动（这是与 WOW64 的根本区别）

## 六、驱动模型与安全增强

### 6.1 强制驱动签名

NT 5.2 x64 是微软**首个强制要求内核驱动必须经过数字签名**的 Windows 版本，未签名驱动无法加载。这一策略在 32 位版本中并未强制执行。

### 6.2 PatchGuard（内核补丁保护）

NT 5.2 x64 引入了 **KPP（Kernel Patch Protection）**，即 PatchGuard：

* 周期性校验关键内核结构（SSDT、IDT、GDT、内核代码段等）
* 一旦检测到被篡改，立即触发 `0x109 CRITICAL_STRUCTURE_CORRUPTION` 蓝屏
* 目的是阻止 Rootkit 和第三方安全软件 Hook 内核

### 6.3 NX（No-Execute）位

x86-64 架构的页表项包含 **XD/NX 位**，NT 5.2 x64 充分利用此特性，将数据页标记为不可执行，大幅提高了对缓冲区溢出攻击的抵抗能力。

### 6.4 引导链（与后续 Windows 区分）

Windows XP x64 仍属于 **NTLDR 时代**的启动路径（**不是** Windows Vista 起的 **Bootmgr + winload**）。引导加载程序负责把系统带入 **长模式** 并加载 **64 位 `ntoskrnl.exe`** 等核心映像；具体阶段划分与 32 位 XP **概念相近**，但存在 **AMD64 与 ACPI/SSE2** 等硬件前提。

**本仓库**：`boot/entry.S` + Multiboot2 + `src/kernel/entry.zig` 提供的是 **独立实验内核** 的入口，**不** 复刻 NTLDR/SETUP 二进制行为；目的见 `docs/KERNEL_AND_STACK_CN.md` 与 `docs/REPOSITORY_LAYOUT_CN.md`。

### 6.5 开发与调试生态（论坛常问方向）

| 主题 | 说明 |
|------|------|
| **DDK / 早期 WDK** | 面向 NT 5.2 x64 的驱动开发需使用 **对应年代** 的驱动套件与目标 OS 的符号；MSDN/OSR 讨论区常有「头文件与构建环境」话题。 |
| **内核调试** | **KD / WinDbg** 双机内核调试仍是分析 KMD、IRP 与崩溃转储的主流方式；符号服务器与旧版系统 PDB 获取方式见社区帖。 |
| **签名与加载策略** | x64 下 **未签名 KMD 默认无法加载**；与 PatchGuard 一起，是论坛里 **补丁、测试签名、开发机策略** 讨论的热点（生产环境请遵守许可与安全策略）。 |

## 七、与 NT 5.1（Windows XP 32 位）的主要差异总结

| 特性         | NT 5.1 x86      | NT 5.2 x64 |
| ---------- | --------------- | ---------- |
| 用户态地址空间    | 最大 3 GB         | **8 TB**   |
| 最大物理内存     | 4 GB（PAE 64 GB） | **128 GB** |
| 页表层级       | 2 级（+ PAE 3 级）  | **4 级**    |
| 32 位兼容     | 原生              | WOW64 翻译层  |
| 16 位应用     | 支持（NTVDM）       | **不支持**    |
| 驱动签名       | 可选              | **强制**     |
| PatchGuard | 无               | **有**      |
| NX 位强制     | 软件模拟            | **硬件原生**   |

---

NT 5.2 的 64 位内核架构本质上是微软将经过 Windows NT 多年积累的成熟分层设计平滑迁移到 AMD64 平台的产物，同时借助新硬件特性在安全性上实现了代际跃升，为后续 Windows Vista（NT 6.0）的全面 64 位化奠定了基础。
